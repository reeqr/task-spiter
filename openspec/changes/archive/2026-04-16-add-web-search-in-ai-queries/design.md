## Context

当前项目的 AI 请求由前端直接调用模型 Completion 接口，核心入口在 `src/utils/api.ts` 的 `breakdownTask`、`breakdownConcept`、`consultAI`、`queryKnowledgeAI`、`queryExamAnglesAI` 与 `requestQueryAnswer`。现状下回答主要依赖模型参数知识和提示词，不具备稳定的实时检索能力，也没有统一的来源追溯机制。  
本次变更需要在不破坏现有多 provider 兼容结构的前提下，为所有 AI 场景增加“全局联网搜索增强”，并确保用户可感知联网状态与结果来源。

## Goals / Non-Goals

**Goals:**
- 为所有 AI 请求提供统一联网搜索能力（包括任务拆解、概念拆解、术语问答）。
- 通过一个全局开关与参数配置控制联网行为，支持关闭与降级。
- 在回答结果中保留可追溯来源信息（至少链接与发布时间）。
- 对不支持 `web_search` 工具调用的模型/provider 自动回退到普通问答，保证可用性。

**Non-Goals:**
- 不在本次变更中重构整体 provider 抽象层或新增后端代理服务。
- 不要求所有 provider 都实现同等联网能力，仅保证“支持时启用，不支持时降级”。
- 不在本次变更中引入复杂检索排序、重排或多轮检索策略。

## Decisions

### Decision 1: 采用“模型原生工具调用”作为首选路径
在支持的 provider（优先智谱）中，调用 Completion 接口时附带 `tools.web_search` 参数，让模型在回答阶段自动检索并整合结果。  
**Rationale:** 现有架构以前端直连模型为主，此方案改动最小且能快速落地。  
**Alternatives considered:**  
- 先调用独立搜索 API 再拼接上下文：可控性更高，但需要更大改动和额外密钥管理策略。  
- 仅通过提示词声明“请联网”：不可验证、不可控，不满足产品能力要求。

### Decision 2: 使用单一全局开关控制联网能力
在模型管理中新增一个“联网搜索开关”和基础参数（条数、域名过滤、时间过滤、内容粒度）。开关开启后，所有 AI 请求默认尝试注入 `tools.web_search`。  
**Rationale:** 联网会带来成本、延迟与稳定性波动，必须让用户可控。  
**Alternatives considered:**  
- 按查询类型分别开关：配置复杂，用户理解成本更高。  
- 每次弹窗二次确认：交互成本高，影响连续追问体验。

### Decision 3: 用“能力表 + 运行时探测”判断支持性并自动降级
系统先基于 provider 能力表判断是否可尝试 `web_search`，并在运行时通过错误响应进一步探测。若出现不支持工具调用、参数被拒绝或联网检索失败，系统不终止流程，自动改为普通请求并提示“已降级”。  
**Rationale:** 保持主流程稳定优先，避免因联网能力不稳定导致功能不可用。  
**Alternatives considered:**  
- 直接报错中断：可见性高但体验差。  
- 静默失败不提示：用户无法理解结果是否联网增强，信任感下降。

### Decision 4: 回答展示中要求来源可追溯
联网增强模式下，回答需包含引用来源（链接、发布时间）或独立来源区块，便于用户核验。  
**Rationale:** 联网功能的价值之一是“可验证”，否则难以区分与普通生成回答。  
**Alternatives considered:**  
- 不展示来源，仅内部使用：实现简单但不满足可追溯目标。

## Risks / Trade-offs

- [Provider 参数不一致] → 使用 provider 能力表定义静态支持性，并通过运行时探测动态拉黑不支持 provider。  
- [联网增加响应时延] → 开关默认可关闭，限制默认检索条数并允许用户调小。  
- [来源质量参差] → 支持域名过滤与时间过滤，并在提示词中要求标注来源时间。  
- [兼容性回归风险] → 为全局开关关闭、联网成功、联网失败降级三类路径分别增加验证用例。

## Migration Plan

1. 先发布配置项与请求构造改动，默认保持关闭。  
2. 在小范围手动验证后，确认全局开关开启时仅支持 provider 会联网，不支持 provider 自动降级。  
3. 若出现异常（高失败率/延迟不可接受），可通过配置立即关闭联网开关回滚到旧行为。  

## Open Questions

- 来源展示是否需要结构化卡片（标题+链接+时间），还是先以 Markdown 引用列表实现？  
- 对追问场景，是否沿用首次检索结果，还是每轮追问都重新检索？
