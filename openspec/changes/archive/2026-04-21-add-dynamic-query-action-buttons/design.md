## Context

当前查询入口由组件硬编码：术语卡片固定显示“解释”“出题角度”，概念标题区固定显示同类按钮；提示词模板也按固定键维护。该实现导致新增“出题角度变体”必须改代码和配置结构，无法支持用户按学习场景快速调整按钮文案与提示词。  
本次变更需要跨越配置层（模板与动作定义）、API 组装层（按动作选择模板）、UI 渲染层（按钮动态渲染）和会话层（追问按动作隔离），属于跨模块行为改造。

## Goals / Non-Goals

**Goals:**
- 提供动作配置模型，支持多个查询按钮并可设置文案、启用状态、排序、首轮模板和追问模板。
- 术语与概念两处入口统一按配置渲染按钮，点击后按 `actionId` 发起查询。
- 追问弹窗在保留现有流式输出与来源展示能力下，新增按 `actionId` 选择上下文模板。
- 兼容既有本地配置，确保升级后至少存在默认动作且可立即使用。

**Non-Goals:**
- 本次不引入服务端持久化，配置仍保存在浏览器本地。
- 本次不实现复杂的动作可视化编排（拖拽排序、分组、导入导出）。
- 本次不改动 AI 提供商管理与联网搜索主流程。

## Decisions

### 决策 1：新增 `queryActions` 配置并与模板同源持久化
- **选择**：在现有提示词配置模块中新增 `queryActions` 数组，每项包含 `id`、`label`、`enabled`、`sort`、`queryTemplate`、`followupTemplate`。
- **原因**：与现有模板读写逻辑共用存储入口，迁移与回滚成本最低。
- **备选方案**：单独维护动作配置 key。被拒绝原因是状态分裂，容易出现模板与按钮映射不一致。

### 决策 2：统一查询入口函数签名为 `actionId` 驱动
- **选择**：将“解释/出题角度”固定接口抽象为通用查询入口，由 `actionId` 决定 prompt 构建逻辑。
- **原因**：减少分支重复，新增动作无需再新增并行函数。
- **备选方案**：保留多套函数并在每个函数内读取动作配置。被拒绝原因是扩展动作数量后维护成本线性上升。

### 决策 3：会话状态加入 `actionId` 和 `actionLabel`
- **选择**：弹窗状态记录当前动作标识与显示名，用于标题和追问模板路由。
- **原因**：同一术语可在不同动作下建立独立语义会话，避免上下文污染。
- **备选方案**：仅按术语隔离会话。被拒绝原因是不同动作会共享历史，导致追问语义偏移。

### 决策 4：提供前向兼容迁移函数
- **选择**：加载配置时检测旧结构，自动补齐默认动作（如“解释”“出题角度”）并映射旧模板到对应默认动作。
- **原因**：保证无痛升级，不要求用户手动重配。
- **备选方案**：版本升级后重置全部模板。被拒绝原因是会丢失用户已有自定义模板。

## Risks / Trade-offs

- [风险] 动作被全部禁用导致查询入口消失 → **缓解**：保存前校验至少保留一个启用动作，或在运行时回退到默认动作。
- [风险] 用户模板变量缺失导致 prompt 质量下降 → **缓解**：保存时进行变量提示与基础校验，不阻断但提醒。
- [风险] 旧配置迁移映射错误影响按钮行为 → **缓解**：迁移逻辑单元测试覆盖“空配置/旧配置/部分字段缺失”场景。
- [权衡] 通用化后代码抽象层次上升，短期可读性略降 → **缓解**：保留简洁命名和最小注释，避免过度封装。
