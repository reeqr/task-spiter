## Context

当前 `ConceptBreakdownPage` 已具备“解释”和“出题角度”两类查询入口，并分别以弹窗展示 AI 返回文本，但弹窗状态仅保存单次 `answer`。本次变更需要在不引入后端会话服务的前提下，为两个弹窗提供前端多轮会话能力，并保证关闭即清空、两类弹窗上下文互不污染。

## Goals / Non-Goals

**Goals:**
- 在“解释”与“出题角度”弹窗内支持多轮追问，并以会话流展示
- 统一发送行为为 Enter 发送，发送期间防重复触发
- 关闭弹窗后立即清空当前会话与输入状态
- 保证解释会话与出题角度会话独立维护，不共享历史
- 将追问提示词纳入现有模板体系（加载/保存/恢复默认）

**Non-Goals:**
- 不新增后端会话持久化或会话 ID 机制
- 不实现跨刷新恢复会话
- 不改造术语按钮总体布局或新增查询类型

## Decisions

### 1) 采用前端内存会话，不引入服务端会话 ID
- **Decision**: 在页面状态中分别维护 `knowledgeChat` 与 `examAngleChat` 的消息数组与输入值，每次追问将最近会话片段拼接到请求上下文
- **Rationale**: 现有架构为前端直连模型，前端会话可快速落地且改动范围可控
- **Alternative Considered**: 引入后端会话服务；可扩展性更好，但当前实现成本高且超出本次范围

### 2) 会话生命周期采用“弹窗关闭即清空”
- **Decision**: 在两个弹窗的 `onCancel` 中统一重置该弹窗会话状态（消息、输入、发送态）
- **Rationale**: 与你确认的学习模式一致，避免历史上下文残留引发误导
- **Alternative Considered**: 按术语缓存会话；虽可续聊，但会增加状态复杂度与上下文污染风险

### 3) 发送交互统一为 Enter 发送
- **Decision**: 输入框监听 Enter 触发发送，Shift+Enter 预留为换行（若多行输入被启用），发送中禁用再次提交
- **Rationale**: 减少点击成本，符合高频追问场景
- **Alternative Considered**: 仅保留“发送按钮”；学习流畅度略低

### 4) 保持解释与出题角度完全独立会话
- **Decision**: 两个弹窗使用独立状态容器、独立 API 方法与独立提示词模板
- **Rationale**: 两类问题语义不同，独立上下文可避免提示词串扰
- **Alternative Considered**: 共享一套会话状态并通过 queryType 区分；实现可复用但维护风险更高

## Risks / Trade-offs

- [Risk] 多轮历史导致 token 增长与响应变慢 → Mitigation: 仅携带最近 N 轮上下文并预留截断策略
- [Risk] Enter 发送与输入法回车冲突 → Mitigation: 仅在非 composition 状态触发发送
- [Risk] 前端状态分支增多带来维护负担 → Mitigation: 提炼通用对话状态结构与发送流程工具函数
- [Risk] 模板字段扩展引发旧缓存兼容问题 → Mitigation: `loadPromptTemplates` 为新增字段提供默认值兜底

## Migration Plan

1. 扩展提示词模板类型与默认模板，加入解释追问/出题角度追问模板并做旧缓存兼容
2. 扩展 API 查询方法，支持接收历史消息与当前追问内容
3. 重构两个结果弹窗状态结构，新增消息流渲染、输入框与 Enter 发送
4. 在弹窗关闭流程中统一执行会话清空，验证“关闭即清空”
5. 回归验证独立会话：解释与出题角度互不影响，不同术语之间不串上下文
