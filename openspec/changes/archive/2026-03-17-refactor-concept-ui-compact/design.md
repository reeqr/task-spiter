## Context

当前的概念拆解界面采用两列布局,左侧显示专业术语列表,右侧显示知识考点列表。每个卡片包含多行内容(标题、定义/描述),且卡片间距较大。用户反馈希望:
1. 提高信息密度,在一屏内看到更多内容
2. 将考点改为按需查询,而不是始终占据屏幕空间
3. 移除最外层概念在列表中的冗余显示

**现有实现:**
- `ConceptBreakdownPage`: 使用 Space 组件分隔术语和考点两个区域
- `TerminologyList`: 渲染术语卡片,支持递归展开
- `KnowledgePointList`: 渲染考点卡片,支持递归展开
- 每个卡片使用 Ant Design Card 组件,内边距 20px,卡片间距 middle

**技术栈:**
- React 19 + TypeScript
- Ant Design 6 (Card, Space, Button, Modal)
- 现有的递归拆解和状态管理系统

## Goals / Non-Goals

**Goals:**
- 将考点从固定展示改为按钮触发的模态框展示
- 压缩列表项为单行显示(术语名+定义在同一行,文本溢出省略)
- 减小卡片内边距和列表间距,提高信息密度
- 移除最外层概念在列表中的显示,改为页面顶部固定区域
- 保持递归拆解功能和二次元风格不变
- 保持现有的展开/收起和继续拆解功能

**Non-Goals:**
- 不改变 AI 拆解逻辑和 API 调用方式
- 不修改数据结构(Term, KnowledgePoint, ConceptBreakdown)
- 不实现考点的持久化缓存
- 不添加考点搜索或过滤功能

## Decisions

### 决策 1:考点查询方式
**选择:** 为每个术语添加"查询考点"按钮,点击后在 Modal 中显示该术语的相关考点

**理由:**
- Modal 不占用主界面空间,提高列表信息密度
- 用户可以按需查看考点,减少认知负担
- Modal 可以复用现有的 KnowledgePointList 组件渲染考点

**实现方式:**
```typescript
// 在 TerminologyList 中添加
<Button
  size="small"
  icon={<QuestionCircleOutlined />}
  onClick={() => onQueryKnowledge(term.id, term.name)}
>
  考点
</Button>

// 在 ConceptBreakdownPage 中
const [knowledgeModal, setKnowledgeModal] = useState<{
  visible: boolean;
  term: string;
  points: KnowledgePoint[];
}>({ visible: false, term: '', points: [] });
```

**备选方案:**
- Popover 组件:空间太小,无法展示完整考点列表
- Drawer 组件:占用整个侧边,过于突兀
- 内联展开:与列表混在一起,不符合紧凑布局目标

### 决策 2:单行布局实现
**选择:** 使用 flexbox 布局,术语名和定义在同一行,不截断文本

**样式方案:**
```css
.compact-term-item {
  display: flex;
  align-items: center;
  gap: 12px;
}

.term-name {
  flex-shrink: 0;
  font-weight: bold;
  min-width: 100px;
}

.term-definition {
  flex: 1;
}
```

**理由:**
- 通过修改AI提示词控制定义长度(20-30字以内),无需截断
- Flexbox 布局简单灵活,易于控制对齐和间距
- 保持完整信息,用户无需悬停查看

**备选方案:**
- 截断+省略号:用户体验不好,需要额外操作
- 多行显示:不符合紧凑布局目标
- 固定宽度:不适配不同屏幕尺寸

### 决策 3:AI提示词优化
**选择:** 修改概念拆解的AI提示词,要求生成简短的术语定义

**提示词要求:**
- 术语定义控制在20-30字以内
- 考点描述控制在30-40字以内
- 保持语义完整性,不要过于简略

**理由:**
- 从源头控制内容长度,比UI截断更优雅
- 简短定义更适合快速浏览
- 用户可以通过"继续拆解"或"查询考点"获取更多详情

**实现方式:**
```typescript
const prompt = `请分析以下概念，并提取其中的专业术语。
概念：${concept}
要求：
1. 提取 5-10 个相关的专业术语
2. 每个术语的定义控制在20-30字以内,简洁明了
3. 提取 5-10 个知识考点(用于按钮查询)
4. 每个考点的描述控制在30-40字以内
...
```

### 决策 4:移除分组标题
**选择:** 移除"专业术语"、"知识考点"等分组标题

**理由:**
- 主列表只显示术语,无需"专业术语"标题
- 递归子项直接展示,不需要分组标题
- 减少视觉干扰,提高信息密度

**实现方式:**
- 在 TerminologyList 中移除 "专业术语" 和 "知识考点" 的 Text 组件
- 子项直接渲染,不添加分组文字

### 决策 5:间距和内边距调整
**选择:**
- 卡片内边距从 20px 减小到 12px
- 列表间距从 "middle"(16px) 减小到 "small"(8px)
- 行高从默认 1.5 减小到 1.2

**理由:**
- 在不影响可读性的前提下,最大化信息密度
- 8-12px 是合理的紧凑间距,不会显得拥挤
- 行高 1.2 仍然符合可读性标准

**测试方案:**
- 在不同屏幕尺寸下验证可读性
- 确保触摸设备上的可点击区域足够大(至少 44x44px)

## Risks / Trade-offs

### 风险 1:AI生成定义过长
**影响:** 即使提示词要求简短,AI可能仍生成较长定义导致单行显示拥挤

**缓解措施:**
- 在提示词中明确字数限制(20-30字)
- 添加示例定义供AI参考
- 如果定义过长,前端可以选择性换行(在特定断点处)
- 用户可通过"继续拆解"获取更详细信息

### 风险 2:考点按需加载增加交互成本
**影响:** 用户需要额外点击才能查看考点,可能降低考点的可见性

**缓解措施:**
- 按钮设计显眼,使用图标 + 文字
- Modal 加载快速(复用现有 AI 拆解结果)
- 考虑后续添加"展开所有考点"的批量操作

### 风险 3:紧凑布局在小屏幕上显示问题
**影响:** 手机等小屏设备上单行显示可能过于拥挤

**缓解措施:**
- 使用响应式设计,小屏幕下恢复多行布局
- 术语名最小宽度使用百分比而非固定值
- 测试 iPad 和手机尺寸

### Trade-off:信息密度 vs 考点可见性
**取舍:** 选择更高的信息密度,将考点改为按需查询

**接受理由:** 用户明确要求压缩显示以便浏览更多内容,且提供了 Modal 作为补充查看方式
